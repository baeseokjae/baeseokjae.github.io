How Do You Install and Configure the Pulumi Python SDK?#

Installing Pulumi Python SDK requires the Pulumi CLI plus the Python provider packages relevant to your cloud targets. The CLI handles state management, previews, and deployments. Python packages handle the resource definitions.

# Install Pulumi CLI (Linux/Mac)
curl -fsSL https://get.pulumi.com | sh

# Verify installation
pulumi version  # Should show 4.x

# Create a new Python project
mkdir my-ai-infra && cd my-ai-infra
pulumi new python -y

# Install cloud providers
pip install pulumi pulumi-aws pulumi-gcp pulumi-azure-native pulumi-kubernetes

After creating a project, you’ll have __main__.py (your infra code), Pulumi.yaml (project config), and a Pulumi.dev.yaml (stack-specific config). Authenticate to your cloud provider using standard methods — aws configure, gcloud auth application-default login, or az login. Pulumi uses the same credentials as the SDK or CLI for that provider.

For teams using Pulumi Cloud (the managed backend), pulumi login stores state remotely. Self-hosted teams can point Pulumi at an S3 bucket or Azure Blob:

pulumi login s3://my-pulumi-state-bucket

What Are the Core Pulumi Concepts Every AI Engineer Needs?#

Pulumi’s programming model maps cleanly onto concepts AI/ML engineers already know: programs, functions, and typed return values.

Resources are cloud objects — an EC2 instance, a GKE node pool, a SageMaker endpoint. You create them by instantiating Python classes:

import pulumi_aws as aws

# Creates a GPU-optimized EC2 instance
gpu_instance = aws.ec2.Instance("llm-training-node",
    instance_type="p3.2xlarge",
    ami="ami-0c02fb55956c7d316",  # Deep Learning AMI
    tags={"Purpose": "llm-training", "Team": "ml-platform"}
)

Outputs are the typed return values that Pulumi resolves after deployment — IP addresses, ARNs, endpoint URLs. You chain them with .apply() to use them as inputs to other resources:

# Use the instance's public IP in a Route53 record
endpoint_record = aws.route53.Record("model-endpoint",
    zone_id=hosted_zone.id,
    name="inference.internal",
    type="A",
    ttl=300,
    records=[gpu_instance.public_ip]
)

Stacks are independent deployments of the same program — dev, staging, prod. Each stack has its own state and config. You switch with pulumi stack select prod.

Config stores stack-specific values without hardcoding:

config = pulumi.Config()
model_size = config.get("model_size") or "7b"
gpu_count = config.get_int("gpu_count") or 2

How Do You Provision GPU Clusters on AWS with Pulumi Python?#

GPU cluster provisioning for LLM training on AWS with Pulumi Python means combining EC2 P-series or G-series instances, placement groups for low-latency node communication, and EKS for orchestration. The key advantage over Terraform is that your cluster configuration is real Python — you can loop over instance types, calculate costs programmatically, or pull model size parameters from a config file.

A complete AWS GPU cluster for LLM training using Pulumi Python looks like this: you define a VPC with private subnets, create a placement group with cluster strategy for RDMA networking between nodes, provision a launch template with the Deep Learning AMI and NVIDIA driver bootstrapping in user data, attach an EKS-managed node group targeting p3.16xlarge (8x NVIDIA V100) or g4dn.12xlarge (4x T4) instances, and configure cluster autoscaler to scale between 0 and N nodes based on GPU utilization. For training jobs that run nightly, scaling to zero when idle cuts costs by 80–90%. Pulumi 4.0+’s Incremental State Processing reduces deployment times by 60% for infrastructures with 1,000+ resources — critical when your ML platform spans dozens of node groups.

import pulumi_aws as aws
import pulumi_eks as eks

# GPU-optimized EKS cluster for LLM workloads
gpu_cluster = eks.Cluster("llm-training-cluster",
    instance_type="p3.16xlarge",
    desired_capacity=0,
    min_size=0,
    max_size=8,
    gpu=True,
    node_associate_public_ip_address=False,
    tags={"CostCenter": "ml-platform", "Env": "training"}
)

# Export kubeconfig for kubectl access
pulumi.export("kubeconfig", gpu_cluster.kubeconfig)

For multi-node training with Elastic Fabric Adapter (EFA), add EFA network interfaces and the p4d.24xlarge instance type (8x A100 GPUs with 400 Gbps EFA networking):

efa_launch_template = aws.ec2.LaunchTemplate("efa-p4d",
    instance_type="p4d.24xlarge",
    network_interfaces=[{
        "interfaceType": "efa",
        "deviceIndex": 0,
    }]

How Do You Deploy Amazon SageMaker Endpoints with Pulumi Python IaC?#

SageMaker endpoint management with Pulumi Python is the clearest example of the unified Python advantage: your model training code, the SageMaker model registration, and the endpoint infrastructure all live in the same language. You use pulumi_aws to provision IAM roles (with least-privilege policies for SageMaker to access S3), create a aws.sagemaker.Model resource pointing to your S3 model artifact, define an endpoint configuration with instance type and variant weights, and deploy the endpoint — all in a single Pulumi program you can run with pulumi up.

import pulumi_aws as aws

# IAM role for SageMaker with S3 model access
sagemaker_role = aws.iam.Role("sagemaker-exec-role",
    assume_role_policy=json.dumps({
        "Version": "2012-10-17",
        "Statement": [{"Effect": "Allow", "Principal": 
            {"Service": "sagemaker.amazonaws.com"}, 
            "Action": "sts:AssumeRole"}]
    })
)

aws.iam.RolePolicyAttachment("sm-s3-access",
    role=sagemaker_role.name,
    policy_arn="arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess"
)

# Register HuggingFace model artifact
llm_model = aws.sagemaker.Model("llama3-8b-model",
    execution_role_arn=sagemaker_role.arn,
    primary_container=aws.sagemaker.ModelPrimaryContainerArgs(
        image="763104351884.dkr.ecr.us-east-1.amazonaws.com/huggingface-pytorch-tgi-inference:2.1.1-tgi2.0.0-gpu-py310",
        model_data_url="s3://my-models/llama3-8b/model.tar.gz",
        environment={"HF_MODEL_ID": "meta-llama/Llama-3-8B-Instruct"}
    )
)

# Endpoint configuration (ml.g5.2xlarge = 1x A10G GPU)
endpoint_config = aws.sagemaker.EndpointConfiguration("llama3-config",
    production_variants=[aws.sagemaker.EndpointConfigurationProductionVariantArgs(
        variant_name="primary",
        model_name=llm_model.name,
        initial_instance_count=1,
        instance_type="ml.g5.2xlarge",
    )]
)

endpoint = aws.sagemaker.Endpoint("llama3-endpoint",
    endpoint_config_name=endpoint_config.name
)

pulumi.export("endpoint_name", endpoint.name)

CloudWatch alarms for latency and error rate can be added to the same file, ensuring observability is never decoupled from deployment.

How Do You Build Multi-Cloud AI Pipelines with Shared Python Code?#

Multi-cloud AI pipeline management with Pulumi Python means writing a single Python abstraction layer that deploys inference infrastructure to AWS SageMaker, GCP Vertex AI, or Azure ML — selecting the target at stack selection time, not at code-writing time. This is impossible in Terraform without complex module hierarchies and provider aliasing. In Python, it’s a class with a cloud-specific implementation and a shared interface.

The pattern: define an abstract ModelEndpoint class in Python, implement AWSModelEndpoint, GCPModelEndpoint, and AzureModelEndpoint subclasses, then instantiate the right one based on pulumi.Config().require("cloud_target"). All three implementations emit the same Pulumi Output (endpoint_url) that downstream resources — DNS records, API gateway routes, monitoring dashboards — consume without knowing which cloud is underneath. This architecture lets one ML platform team support three different business units each with different cloud contracts, using one Pulumi codebase with three stacks: prod-aws, prod-gcp, prod-azure.

config = pulumi.Config()
cloud = config.require("cloud_target")  # "aws", "gcp", or "azure"

if cloud == "aws":
    endpoint = deploy_sagemaker_endpoint(model_config)
elif cloud == "gcp":
    endpoint = deploy_vertex_endpoint(model_config)
elif cloud == "azure":
    endpoint = deploy_azureml_endpoint(model_config)

pulumi.export("inference_url", endpoint.url)

How Does Pulumi ESC Handle Secrets for AI Stacks?#

Pulumi ESC (Environments, Secrets, and Configuration) is a centralized secrets management layer purpose-built for multi-stack, multi-cloud deployments — exactly what AI infrastructure generates. An AI platform typically accumulates dozens of secrets: HuggingFace tokens, OpenAI API keys, database passwords for vector stores, credentials for model registries, and per-environment endpoint URLs. Without centralized management, these scatter across CI environment variables, .env files, and per-stack Pulumi configs. ESC consolidates them into versioned, audited environments with role-based access.

ESC environments are YAML documents stored in Pulumi Cloud that compose secrets from multiple backends (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault) and expose them as unified config to any Pulumi stack — or directly to CLI commands and CI pipelines. For an AI team, this means one ESC environment called ai-platform/prod holds every secret your ML platform needs, and any authorized engineer can pull a scoped subset with esc open ai-platform/prod:

# ESC environment: ai-platform/prod
values:
  huggingface:
    token:
      fn::secret:
        ciphertext: "ZXNjeV..."
  openai:
    api_key:
      fn::fromSecret:
        secret: arn:aws:secretsmanager:us-east-1:123456:secret:openai-key
  model_endpoints:
    sagemaker: "https://runtime.sagemaker.us-east-1.amazonaws.com/endpoints/llama3"
    vertex: "https://us-central1-aiplatform.googleapis.com/v1/projects/..."

Your Pulumi Python program then references ESC config without any hardcoded secrets:

config = pulumi.Config("ai-platform")
hf_token = config.require_secret("huggingface.token")

How Does CrossGuard Policy as Code Work for AI Compliance?#

Pulumi CrossGuard is a Policy as Code framework that runs before every pulumi up, enforcing compliance rules against the planned infrastructure changes. For AI workloads, this means you can programmatically prevent GPU overprovisioning, enforce data residency constraints, require encryption on all model artifact buckets, and cap per-stack GPU spend — all in Python, all enforced at the infrastructure layer rather than relying on manual review.

CrossGuard policies are Python programs that receive resource metadata before deployment and emit pass/warn/fail decisions. An AI platform might enforce: all S3 buckets storing model artifacts must have server-side encryption enabled; GPU instances above a threshold count require a cost-center tag; SageMaker endpoints in prod must use multi-AZ configurations; no training jobs can use regions outside us-east-1 and eu-west-1 for GDPR compliance. These policies run in a separate Policy Pack registered with Pulumi Cloud, and any violation blocks the deployment:

from pulumi_policy import PolicyPack, ResourceValidationPolicy, EnforcementLevel

def gpu_budget_check(args, report):
    if args.resource_type == "aws:sagemaker/endpoint:Endpoint":
        instance_type = args.props.get("productionVariants", [{}])[0].get("instanceType", "")
        if instance_type.startswith("ml.p4") and not args.props.get("tags", {}).get("CostApproved"):
            report.fail("P4 SageMaker instances require CostApproved=true tag")

PolicyPack("ai-governance",
    policies=[ResourceValidationPolicy(
        name="gpu-budget-check",
        description="Require cost approval for large GPU instances",
        validate=gpu_budget_check,
        enforcement_level=EnforcementLevel.MANDATORY
    )]
)

How Do You Test Pulumi Python Infrastructure with pytest?#

Testing Pulumi Python infrastructure with pytest covers three layers: unit tests that mock cloud provider calls and assert resource configurations, integration tests that deploy to a real cloud sandbox and verify actual behavior, and preview-mode checks that run pulumi preview in CI and fail on unexpected resource changes. This is the clearest advantage over HCL-based tools — Terraform has no native unit testing that can mock provider calls.

For unit tests, Pulumi provides a pulumi.runtime.set_mocks() API that intercepts all resource creation calls and returns mock outputs. You can then assert that an EKS cluster was created with the right GPU instance type, that encryption is enabled on all S3 buckets, or that a SageMaker endpoint has the right IAM role attached — without making any AWS API calls:

import pytest
import pulumi

class MyMocks(pulumi.runtime.Mocks):
    def new_resource(self, args):
        return [args.name + "_id", args.inputs]
    def call(self, args):
        return {}

pulumi.runtime.set_mocks(MyMocks())

import infra  # Your Pulumi __main__.py

@pytest.mark.asyncio
async def test_gpu_cluster_instance_type():
    instance_type = await infra.gpu_cluster.instance_type.future()
    assert "p3" in instance_type or "g4dn" in instance_type, \
        f"Expected GPU instance, got {instance_type}"

@pytest.mark.asyncio
async def test_model_bucket_encryption():
    encryption = await infra.model_bucket.server_side_encryption_configuration.future()
    assert encryption is not None, "Model bucket must have SSE enabled"

Run with pytest tests/ -v. Integration tests use pulumi up --stack test-ephemeral in CI and clean up with pulumi destroy after assertions.

How Do You Integrate Pulumi into CI/CD for AI Infrastructure Pipelines?#

CI/CD integration for Pulumi Python AI infrastructure connects your Git workflow to automated deployments: preview on pull request, deploy on merge to main, destroy ephemeral stacks after feature branch tests. The recommended pattern uses GitHub Actions with the pulumi/actions action, Pulumi ESC for secrets (so no secrets live in GitHub), and stack-specific config per environment.

# .github/workflows/pulumi-deploy.yml
name: Deploy AI Infrastructure
on:
  push:
    branches: [main]
  pull_request:

jobs:
  preview:
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    steps:
      - uses: actions/checkout@v4
      - uses: pulumi/actions@v5
        with:
          command: preview
          stack-name: ai-platform/staging
          comment-on-pr: true
        env:
          PULUMI_ACCESS_TOKEN: ${{ secrets.PULUMI_ACCESS_TOKEN }}
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}

  deploy:
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: pulumi/actions@v5
        with:
          command: up
          stack-name: ai-platform/prod

For ML teams running nightly training pipelines, a separate workflow triggers pulumi up before the training job (scaling GPU nodes from 0 to N) and pulumi down after it completes (scaling back to 0), reducing idle GPU costs to zero.

Can Pulumi Python manage Kubernetes AI workloads alongside cloud resources?#

Yes. Pulumi’s pulumi_kubernetes provider lets you define Kubernetes Deployments, Services, and custom resources (like NVIDIA GPU Operator CRDs) in the same Python program as your EKS or GKE cluster. Pulumi wires cluster credentials from the cloud provider output directly into the Kubernetes provider, eliminating the manual kubeconfig export step. You can deploy a GKE cluster and a vLLM inference Deployment in a single pulumi up.

How does Pulumi handle Pulumi state for large AI infrastructure?#

Pulumi 4.0+ introduced Incremental State Processing, which reduces deployment times by 60% for infrastructures with 1,000+ resources. State is stored in Pulumi Cloud or a self-hosted backend (S3, GCS, Azure Blob). For large AI platforms with hundreds of resources across multiple stacks, Pulumi recommends stack per environment (dev/staging/prod) and stack per service (training-cluster, inference-endpoints, data-pipeline) to keep individual states small and deployments fast.

Is Pulumi ESC free to use with the Python SDK?#

Pulumi ESC has a free tier for individual developers with limited environments. Team and Enterprise tiers add audit logs, RBAC, and deeper secret backend integrations. The Python SDK integration — config.require_secret() pulling from ESC — works on all tiers. For AI teams managing production secrets across multiple clouds, the Team tier at minimum is recommended.

How do I migrate an existing Terraform AI infrastructure to Pulumi Python?#

Since January 2026, Pulumi natively runs Terraform HCL inside Pulumi programs — you can import existing .tf files and manage them through Pulumi without converting code. For full migration to Python, pulumi convert --from terraform generates Python equivalents of Terraform resources. Run it module by module, validate with pulumi preview, and move state with pulumi import for resources that need to be adopted without recreation.

Does Pulumi Neo work with private cloud or on-premises AI infrastructure?#

Pulumi Neo currently works with Pulumi Cloud-connected deployments and requires outbound internet access to the Pulumi control plane. For air-gapped or on-premises AI infrastructure (e.g., private GPU clusters managed by vSphere or OpenStack), Neo is not available. The underlying Pulumi Python SDK works offline with a self-hosted state backend. Pulumi has indicated private/on-premises Neo support is on the 2026 roadmap for Enterprise customers.
