Before Portkey, teams managing multi-provider AI stacks wrote thousands of lines of custom retry logic, provider-specific error handling, and cost monitoring. How to Get Started with Portkey AI Gateway (Quickstart)#

Getting started with Portkey takes under 10 minutes for most developers — you change two lines of code in your existing OpenAI SDK setup and immediately gain access to routing, logging, and fallback capabilities. The gateway is OpenAI API-compatible, meaning any tool or framework that accepts an openai.OpenAI(base_url=...) argument works without modification. You do not need to learn a new SDK; Portkey acts as a transparent proxy. The managed cloud endpoint is https://api.portkey.ai/v1 and requires only a Portkey API key alongside your provider credentials. For self-hosted deployments, the Docker image adds roughly the same migration path — just point your base URL at your local container. The quickstart below uses the Python SDK, but the same pattern works with Node.js, the REST API directly, or any HTTP client.

Install and configure:

pip install portkey-ai openai

from openai import OpenAI
from portkey_ai import PORTKEY_GATEWAY_URL, createHeaders

client = OpenAI(
    api_key="your-openai-api-key",
    base_url=PORTKEY_GATEWAY_URL,
    default_headers=createHeaders(
        api_key="your-portkey-api-key",
        provider="openai"
    )
)

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello, world!"}]
)
print(response.choices[0].message.content)

This single change routes all your OpenAI requests through Portkey, enabling logging, rate limit handling, and the ability to add fallbacks without further code changes.

Routing to 1,600+ LLMs — How Model Selection Works#

Portkey’s model routing works through Virtual Keys and Config objects — two concepts that decouple your application code from specific provider credentials and model names. A Virtual Key is a Portkey-managed alias for a provider API key; you store real credentials in Portkey’s vault and reference them by a stable identifier like openai-prod or anthropic-backup. A Config is a JSON document that defines routing strategy — which Virtual Keys to use, in what order, with what weights, and under what conditions. Your application references the Config ID rather than hardcoding provider logic. This separation means you can rotate credentials, add providers, or change routing weights in the Portkey dashboard without touching application code.

Portkey supports 1,600+ models across categories: language (GPT-5, Claude Opus 4.7, Gemini 3.1 Pro, Mistral Large 3, Llama 4 Scout), vision (GPT-4o, Gemini Flash 2.5), audio (Whisper, ElevenLabs), image generation (DALL-E 4, Stable Diffusion 4), and embedding models (text-embedding-3-large, Voyage AI). The routing config specifies provider and model for each target, and Portkey translates request/response formats between providers automatically. Switching from GPT-4o to Claude Opus 4.7 requires changing one field in the config, not rewriting SDK calls.

from portkey_ai import Portkey

portkey = Portkey(api_key="your-portkey-api-key")

# Route to Anthropic Claude using a Virtual Key
response = portkey.with_options(
    config={
        "provider": "anthropic",
        "api_key": "anthropic-virtual-key"
    }
).chat.completions.create(
    model="claude-opus-4-7",
    messages=[{"role": "user", "content": "Explain load balancing"}]
)

Setting Up Fallbacks: Keep Your App Running During Provider Outages#

Portkey fallbacks are the single highest-value feature for production AI applications — they automatically retry failing requests through an ordered list of alternative providers or models, keeping your app running when a primary provider goes down. A fallback config is an object with strategy: { mode: "fallback" } and a list of targets; Portkey tries each target in order and stops at the first success. Triggers include HTTP 5xx errors, request timeouts, quota-exceeded (429) responses, and content filter flags. A cascade from OpenAI to Anthropic to a self-hosted Llama model can be defined in 20 lines of JSON and applied to every request in your stack via a single Config ID. In production, teams typically configure a 3-tier chain: primary model (latest capable model), secondary (same provider, cheaper model), and tertiary (different provider, different region). This pattern eliminates the single-provider dependency that causes 2-4 hour outages to ripple through AI-dependent features.

fallback_config = {
    "strategy": {
        "mode": "fallback"
    },
    "targets": [
        {
            "provider": "openai",
            "api_key": "openai-virtual-key",
            "override_params": {"model": "gpt-4o"}
        },
        {
            "provider": "anthropic",
            "api_key": "anthropic-virtual-key",
            "override_params": {"model": "claude-sonnet-4-6"}
        },
        {
            "provider": "groq",
            "api_key": "groq-virtual-key",
            "override_params": {"model": "llama-4-scout"}
        }
    ]
}

portkey = Portkey(api_key="your-portkey-api-key", config=fallback_config)

response = portkey.chat.completions.create(
    model="gpt-4o",  # overridden per target
    messages=[{"role": "user", "content": "Generate a summary"}]
)

Configuring Retry Behavior Within Fallbacks#

Each target in a fallback chain can have its own retry settings. You can specify retry with attempts (1–5) and on_status_codes to control exactly when Portkey retries versus escalates to the next fallback target. For flaky providers, 2 retries with exponential backoff before escalating is a common pattern. For hard quota errors (429), skipping retries and going straight to the next provider is usually correct.

{
    "strategy": {"mode": "fallback"},
    "targets": [
        {
            "provider": "openai",
            "api_key": "openai-virtual-key",
            "retry": {
                "attempts": 2,
                "on_status_codes": [500, 502, 503, 504]
            },
            "override_params": {"model": "gpt-4o"}
        },
        {
            "provider": "anthropic",
            "api_key": "anthropic-virtual-key",
            "override_params": {"model": "claude-opus-4-7"}
        }
    ]
}

Load Balancing Across LLMs and API Keys#

Portkey load balancing distributes requests across multiple API keys or providers using weighted round-robin, automatically normalizing weights so you specify relative ratios rather than percentages. For example, weights of 5, 3, 1 are normalized to 55%, 33%, and 11% — Portkey handles the math. This is particularly useful for two scenarios: rate limit spreading (multiple API keys from the same provider, each with its own RPM/TPM quota) and multi-provider cost optimization (splitting traffic between a premium model for complex tasks and a cheaper model for simple ones). Load balancing configs use strategy: { mode: "loadbalance" } and weight per target. The gateway tracks request counts in memory and rotates targets accordingly. Teams at high volume (>$10k/month LLM spend) commonly use load balancing across 5–10 API keys to stay under per-key rate limits and achieve effective 5-10x throughput without a paid quota increase.

load_balance_config = {
    "strategy": {
        "mode": "loadbalance"
    },
    "targets": [
        {
            "provider": "openai",
            "api_key": "openai-key-1",
            "weight": 5,
            "override_params": {"model": "gpt-4o"}
        },
        {
            "provider": "openai",
            "api_key": "openai-key-2",
            "weight": 3,
            "override_params": {"model": "gpt-4o"}
        },
        {
            "provider": "anthropic",
            "api_key": "anthropic-key",
            "weight": 2,
            "override_params": {"model": "claude-sonnet-4-6"}
        }
    ]
}

Combining Load Balancing with Fallbacks#

You can nest a fallback config inside each load-balanced target, so if a particular weighted provider is down, requests automatically cascade to the next fallback before the load balancer sees a failure. This composability is what separates Portkey’s routing from simpler gateways that only support one strategy at a time.

Conditional Routing: Route by User Plan, Model Alias, or Metadata#

Conditional routing lets you send requests to different models or providers based on metadata attached to each request — enabling A/B testing, user-tier routing (free vs. pro plan), canary releases, and region-based compliance routing without changing application code. You pass metadata via request headers or the Portkey SDK’s metadata parameter, and the gateway evaluates routing rules against those values. A typical example: route free-tier users to gpt-4o-mini, pro users to gpt-4o, and enterprise users to claude-opus-4-7, all through the same API call with metadata: {user_tier: "pro"}. Portkey evaluates rules top-down and routes to the first matching target.

conditional_config = {
    "strategy": {"mode": "conditional"},
    "conditions": [
        {
            "query": {"metadata.user_tier": {"$eq": "enterprise"}},
            "then": "enterprise-target"
        },
        {
            "query": {"metadata.user_tier": {"$eq": "pro"}},
            "then": "pro-target"
        }
    ],
    "default": "free-target",
    "targets": [
        {
            "id": "enterprise-target",
            "provider": "anthropic",
            "api_key": "anthropic-vk",
            "override_params": {"model": "claude-opus-4-7"}
        },
        {
            "id": "pro-target",
            "provider": "openai",
            "api_key": "openai-vk",
            "override_params": {"model": "gpt-4o"}
        },
        {
            "id": "free-target",
            "provider": "openai",
            "api_key": "openai-vk",
            "override_params": {"model": "gpt-4o-mini"}
        }
    ]
}

# In your application
portkey = Portkey(api_key="your-portkey-api-key", config=conditional_config)
response = portkey.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Help me analyze this dataset"}],
    metadata={"user_tier": "enterprise"}
)

Composable Configs: Nesting Strategies for Advanced Use Cases#

Portkey’s most powerful feature is config composability — you can nest a fallback strategy inside a load balancer, a conditional router inside a fallback, or any combination that matches your production requirements. This means you can achieve sophisticated routing topologies without custom middleware: load-balance primary traffic across 3 keys, fall back to a different provider if all 3 fail, and apply conditional routing within the fallback based on request metadata. Composable configs replace hundreds of lines of custom routing code with declarative JSON that the Portkey team maintains and optimizes. Enterprise teams at companies like Clearbit and Perplexity use 4–5 layer nested configs to manage routing across dev, staging, and production environments with different reliability and cost targets at each tier.

# Load balance primary traffic, fall back to Anthropic on failure
composed_config = {
    "strategy": {"mode": "fallback"},
    "targets": [
        {
            # First target: load-balanced OpenAI keys
            "strategy": {"mode": "loadbalance"},
            "targets": [
                {
                    "provider": "openai",
                    "api_key": "openai-key-1",
                    "weight": 6,
                    "override_params": {"model": "gpt-4o"}
                },
                {
                    "provider": "openai",
                    "api_key": "openai-key-2",
                    "weight": 4,
                    "override_params": {"model": "gpt-4o"}
                }
            ]
        },
        {
            # Fallback: Anthropic
            "provider": "anthropic",
            "api_key": "anthropic-vk",
            "override_params": {"model": "claude-opus-4-7"}
        }
    ]
}

Guardrails: PII Redaction, Jailbreak Detection, and Output Validation#

Portkey Guardrails is a layer of 50+ automated safety and quality checks that run on every request and response without requiring custom middleware or post-processing code in your application. Guardrails operate at two points: input guardrails intercept the request before it reaches the LLM (blocking or redacting), and output guardrails evaluate the response before it returns to your application (flagging, blocking, or triggering fallback to a safer model). Built-in guardrails include PII redaction (emails, phone numbers, SSNs, credit card numbers), jailbreak and prompt injection detection, JSON schema validation (block malformed structured outputs), RegEx pattern matching, toxicity classification, and custom rule chains. For enterprise teams in regulated industries, Portkey’s SOC2 Type 2, HIPAA, GDPR, and ISO certifications mean guardrails double as compliance controls — PII never reaches the LLM, and audit logs prove it. Enabling a guardrail requires adding one key to your Config JSON; no application code changes needed.

guardrail_config = {
    "provider": "openai",
    "api_key": "openai-vk",
    "input_guardrails": ["portkey-input-guardrail-id"],
    "output_guardrails": ["portkey-output-guardrail-id"],
    "override_params": {"model": "gpt-4o"}
}

This eliminates the common pattern of catching JSON parse errors in application code and retrying manually.

Observability: Logs, Latency, Cost Analytics, and Tracing#

Portkey’s observability layer captures a complete record of every LLM request: provider, model, latency (total, time-to-first-token), prompt tokens, completion tokens, estimated cost, HTTP status, routing decisions, which fallback triggered (if any), and up to 16 custom metadata fields. Every request is assigned a trace ID, and you can group requests by trace, user session, feature, or experiment ID to analyze patterns across dimensions. The Portkey dashboard renders this data as real-time cost trend charts, latency percentile distributions (p50/p95/p99), error rate heatmaps, and model comparison tables. Crucially, observability works across providers with normalized cost metrics — you can compare the cost-per-task of GPT-4o versus Claude Opus 4.7 on your actual workload, not synthetic benchmarks. The Logs API lets you query this data programmatically for integration with existing monitoring stacks (Datadog, Grafana, Splunk). Portkey processes 400 billion+ tokens daily and retains logs for 30 days on the Pro plan, 90 days on Enterprise.

# Attach metadata for observability segmentation

response = portkey.chat.completions.create(

Agent Framework Integrations (LangChain, CrewAI, AutoGen)#

Portkey integrates natively with the major AI agent frameworks — LangChain, CrewAI, AutoGen, LlamaIndex, and OpenAI Agents SDK — so you can add production routing, fallbacks, and observability to existing agent code without restructuring your application. The integration pattern is the same across frameworks: replace the provider’s base client or LLM object with a Portkey-wrapped equivalent. For LangChain, this means using ChatOpenAI with openai_api_base pointing to Portkey’s gateway URL. For CrewAI, you configure the LLM provider through Portkey’s LiteLLM-compatible interface. AutoGen agents accept a config_list where you can point each model config at Portkey. Every agent tool call, chain step, and LLM invocation is then automatically logged, routed, and protected by guardrails — giving you end-to-end observability across multi-agent workflows that would otherwise be opaque.

# LangChain integration
from langchain_openai import ChatOpenAI
from portkey_ai import createHeaders, PORTKEY_GATEWAY_URL

portkey_headers = createHeaders(
    api_key="your-portkey-api-key",
    config="your-config-id",  # Config with fallbacks and load balancing
    metadata={"agent": "research-agent", "version": "v2"}
)

llm = ChatOpenAI(
    api_key="your-openai-api-key",
    base_url=PORTKEY_GATEWAY_URL,
    default_headers=portkey_headers,
    model="gpt-4o"
)

# CrewAI integration
from crewai import LLM

portkey_llm = LLM(
    model="openai/gpt-4o",
    base_url=PORTKEY_GATEWAY_URL,
    api_key="your-openai-api-key",
    extra_headers=portkey_headers
)

Self-Hosted Deployment: Docker, Node.js, and Cloudflare Workers#

Portkey Gateway is fully open-source under Apache 2.0, and self-hosting gives you complete control over data residency, latency, and infrastructure costs at high volumes. The Docker deployment is the most common path for teams that need data to stay within their VPC — no traffic touches Portkey’s managed cloud. The gateway image runs as a stateless proxy, so horizontal scaling behind a load balancer is straightforward. For edge deployments, the Cloudflare Workers build runs the gateway at 300+ PoPs globally with sub-10ms overhead per request. The Node.js deployment is best for teams already running Node-based infrastructure who want to embed the gateway as a library rather than a separate service.

# Docker deployment (simplest path)
docker run -d \
  -p 8787:8787 \
  -e NODE_ENV=production \
  portkeyai/gateway:latest

# Point your client at the local gateway
export PORTKEY_GATEWAY_URL="http://localhost:8787/v1"

# Cloudflare Workers deployment
git clone https://github.com/Portkey-AI/gateway
cd gateway
npm install
npm run deploy  # deploys to your Cloudflare account

FAQ#

The following questions cover the most common points of confusion when evaluating Portkey AI Gateway for production use. Portkey is a purpose-built LLM routing layer — not a generic proxy, not a monitoring-only tool, and not a model marketplace. It sits between your application and LLM providers, applying routing strategies (fallbacks, load balancing, conditional rules), guardrails (PII redaction, jailbreak detection), and observability (per-request cost, latency, token counts) before returning responses to your application. Unlike tools that only add logging (Helicone) or only add routing (OpenRouter), Portkey combines both into a single gateway with enterprise compliance certifications. The answers below address deployment options, SDK compatibility, key storage security, fallback trigger conditions, and scalability — the five questions that determine whether Portkey fits your infrastructure requirements and team capabilities in 2026.

How does Portkey AI Gateway differ from a simple reverse proxy?#

Portkey is not a passthrough proxy — it understands LLM-specific semantics including token counting, provider error codes, streaming responses, and model capability differences. It applies routing logic (fallbacks, load balancing, conditional rules), enforces guardrails on content, normalizes provider response formats, and captures structured observability data per request. A generic reverse proxy like Nginx does none of this; it only forwards bytes.

Can I use Portkey with the official OpenAI Python SDK?#

Yes. Portkey is OpenAI API-compatible. You change two values: set base_url=PORTKEY_GATEWAY_URL and add Portkey-specific headers (API key, config ID) via default_headers. All existing OpenAI SDK calls (chat completions, embeddings, function calling, vision, audio) work without further modification.

Does Portkey store my LLM API keys?#

On the managed cloud, Portkey stores provider API keys in an encrypted vault and issues Virtual Key aliases that your application uses instead. Portkey never logs raw API keys in request logs. On self-hosted deployments, keys are stored and managed entirely within your own infrastructure.

What triggers a fallback in Portkey?#

Fallbacks trigger on: HTTP 5xx responses from providers (server errors), HTTP 429 (rate limit exceeded), request timeouts (configurable per target), and content filter flags. You can also configure specific status codes per target to fine-tune when Portkey escalates versus retries.

Is Portkey suitable for high-volume production workloads?#

Yes. Portkey processes 400 billion+ tokens daily and 2 trillion tokens per day at its peak capacity across its managed cloud infrastructure. The self-hosted Docker deployment is stateless and horizontally scalable. Enterprise customers run Portkey in private VPCs with custom SLAs. High-volume teams (>$10k/month LLM spend) commonly choose self-hosted LiteLLM for pure infrastructure reasons, but Portkey’s managed cloud handles enterprise-scale traffic reliably.
